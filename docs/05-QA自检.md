## 四、安全验证

### 已验证的安全机制

1. **JWT 认证** ✅
   - 无 token → 401
   - 无效 token → 401
   - 有效 token → 200，且 `req.user.id` 来自 JWT 解码

2. **越权防护** ✅
   - 所有数据操作都带 `WHERE user_id = ?` 过滤
   - 即使知道别人的 project_id/module_id/task_id 也无法访问或修改
   - 越权操作返回 404（不泄露资源是否存在）

3. **注册限流** ✅
   - 同一 IP 每小时最多 3 次注册
   - 超过返回 429

4. **登录限流** ✅
   - 5 次失败后锁定 15 分钟
   - 成功登录清空失败计数

5. **Demo 接口限流** ✅
   - 每分钟 3 次，每小时 10 次
   - 防 LLM 额度被刷

6. **输入校验** ✅
   - 密码长度 ≥6
   - 全局输入长度限制（username≤50, text≤2000 等）
   - demo name ≤100 字符

7. **JWT Secret 启动校验** ✅
   - 生产环境（`NODE_ENV=production`）没设或长度<32 → 拒绝启动
   - 开发环境使用默认值 → 警告

8. **API Key 上下文隔离** ✅
   - `X-API-Keys` 请求头的 key 存到 `req.apiKeys`，不写 `process.env`
   - 多用户并发不会串号

9. **错误信息脱敏** ✅
   - 生产环境 500 错误只返回"服务器内部错误"
   - 4xx 错误保留具体信息（业务错误，不涉及内部细节）

10. **SQLite 持久化** ✅
    - 支持 `DATABASE_PATH` 环境变量
    - Railway 部署时挂载 `/data` Volume 即可持久化

**理由**：
1. 所有主链路功能测试通过（26/26）
2. 所有关键安全机制已验证生效

### 部署前必做

1. 在 Railway 配置所有环境变量（参考 `.env.example`）
2. 挂载 Persistent Volume 到 `/data`
3. 设置 `DATABASE_PATH=/data/flowsync.db`
4. 设置 `NODE_ENV=production`
5. 设置 `JWT_SECRET`（≥32 字符强密钥）
6. 部署后访问 `/health` 验证

1. 浏览器打开域名，看到 FlowSync 应用
2. 注册账号 → 登录 → 创建项目 → 验证数据持久
